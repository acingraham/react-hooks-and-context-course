To add state to a functional component, we can use the built-in State Hook

useState

  • returns an array, with the first element the current value and the second a function for setting the value
  • takes a single argument for the initial state. This can be any type / value you like.

useState vs setState

  • useState can take any type
  • useState's setter function doesn't merge states. Instead, it replaces the previous state.

If you have multiple pieces of state you want your component to manage, you might think you need to store the state as an object

You can, but you don't have to since useState can be called multiple times within the same component

This has the advantage of allowing you to break out your state logic and share more easily between components

To replace (most) lifecycle methods, we can use the built-in Effect Hook

useEffect

  • Takes a function that is called after each render
  • Runs after all changes have been flushed to the DOM
  • Unlike componentDidMount or componentDidUpdate, effects scheduled with useEffect don’t block the browser from updating the screen

What we've seen so far replaces componentDidMount and componentDidUpdate, but we can also replace componentWillUnmount

useEffect

  • If you return a function, it'll be called prior to re-rendering and on unmount
  • You can provide an array as a second argument to useEffect to control when the effect will run
  • If you pass no argument, it runs every render
  • If you pass an empty array, it'll only run on mount
  • If you pass a non-empty array, it'll only run on mount and when the values within the array change

useEffect

  • Just like with useState, you can call useEffect multiple times within a component
  • This allows you to encapsulate related logic
  • React will apply every effect used by the component, in the order they were specified

Problems Hooks are trying to solve

  • It’s hard to reuse stateful logic between components
  • Lifecycle methods cause related logic to be separated and unrelated logic to be grouped
  • Classes are confusing for humans & computers

So should we rewrite all our components?

Crucially, Hooks work side-by-side with existing code so you can adopt them gradually. There is no rush to migrate to Hooks. We recommend avoiding any “big rewrites”, especially for existing, complex class components. It takes a bit of a mindshift to start “thinking in Hooks”. In our experience, it’s best to practice using Hooks in new and non-critical components first, and ensure that everybody on your team feels comfortable with them. - The React Team
